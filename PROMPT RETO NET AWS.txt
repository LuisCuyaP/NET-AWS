1. CONTEXTO GENERAL DEL RETO
========================

Tengo una arquitectura distribuida basada en eventos con AWS y .NET.

### Solución 1: Appointment.Api
Proyecto principal Web API en .NET.
Características:
- Clean Architecture
- CQRS con MediatR
- EF Core + SQL Server
- Endpoint POST /appointments
- Endpoint GET /appointments/{insuredId}
- Guarda citas inicialmente en estado Pending
- Publica evento AppointmentRequested a SNS
- Tiene un worker/background service que debe consumir la cola appointment-completion y actualizar el estado a Completed en SQL Server

### Solución 2: Appointment.Pe
Lambda .NET para Perú.
Características:
- Consume mensajes desde SQS
- Valida que countryISO = PE
- Persiste resultado en DynamoDB
- Publica evento AppointmentCompleted a EventBridge
- Usa service pattern / capas, no MediatR

### Solución 3: Appointment.Cl
Lambda .NET para Chile.
Características:
- Igual que PE, pero para countryISO = CL
- processedBy = appointment-cl

### Solución 4: infra-pulumi
Proyecto Pulumi con la infraestructura AWS.

========================
2. ARQUITECTURA OBJETIVO
========================

El flujo completo esperado es:

Cliente
-> Appointment.Api
-> SQL Server (Pending)
-> SNS Topic (AppointmentRequested)
-> SQS por país (según countryISO)
-> Lambda PE o CL
-> DynamoDB por país
-> EventBridge (AppointmentCompleted)
-> SQS appointment-completion
-> Appointment.Api background worker
-> SQL Server (Completed)

Este diseño ya estaba definido desde el inicio del reto. El objetivo es mantener coherencia con esta arquitectura y no desviarnos.
